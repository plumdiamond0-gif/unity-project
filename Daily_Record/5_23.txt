업그레이드 패널 UI 관리 방식 
현재 구조 : 
업그레이드 패널(부모) 
-업그레이드할 무기 이미지(자식1) 
-업그레이드할 무기 리스트(자식2) 
-업그레이드하는데 들 가격(자식3) 
-업그레이드 버튼(자식4) 
-업그레이드할 시 증가되는 버프량(자식5) 

리스트에서 업그레이드할 무기 클릭 시, 그 무기의 이미지,가격,버프량이 뜸. 

1,무기 리스트에서 다 관리 
-> 스크립트 수 적음, 매번 데이터 넘겨줄 필요 없음, 
하나의 스크립트가 너무 무거워짐, 기능 복잡
X

2.각 자식마다 클래스 붙여서 부모에 스크립트에서 다 관리 
-> 기능 구분 명확, 관리 스크립트가 가벼워짐,
O
근데 매번 클릭된 무기의 데이터 다 넘겨줘야 함, 스크립트 많아져서 복잡해질 수 있음.(내 생각)
이건 "데이터를 계속 넘겨주는 부담" 수준이 아니야. WeaponData 하나 넘기는 거라 사실 거의 비용도 없고 코드도 깔끔함.(실제)
"부모 = 상태/흐름 관리"
"자식 = UI 표시만 담당"
->
부모가 현재 선택 무기(currentWeapon)를 들고 있음
자식은 UI 표시만 함
자식끼리 직접 참조 안 함
기능 추가 쉬움
스크립트가 많아져도 역할이 명확함

무기 목록 UI 잠금 이지미 관리 :

1.어드레서블로 하나씩 가져와 위치 초기화
-> 잠금 아이콘은 고정 UI 요소라서 리소스 관리 이득이 거의 없거든.

2.자식에 락 이미지 넣고 인스펙터에서 하나씩 집어넣기
똑같은 이미지 10개 넣는다고 해서 메모리가 10배 되는 게 아님.
즉 잠금 스프라이트 파일은 하나만 메모리에 있고, 각 버튼은 "이 이미지 써!" 하고 참조만 하는 거야.
그래서:
❌ 잠금 이미지 파일이 버튼 개수만큼 복제됨
❌ 메모리 10배 증가
⭕ 스프라이트 하나를 여러 UI가 공유


GetChild(0) 같은 건 나중에 구조 바뀌면 터질 수 있음:
lockImage = transform.GetChild(1);
이런 건 유지보수할 때 은근히 골치 아파져.

무기 목록에서 버튼 눌렸을 때 다른 UI들에서 이미지 보여주는 거 어디서 실행?:
눌린 무기 타입만 전해줄거면 버튼에서 바로 전체 관리 패널에 함수 실행하라 신호 보내기.

버튼 자체 반응 → 버튼

클릭 소리
눌림 애니메이션
마우스 호버 효과
버튼 비활성화 시 색 변경

선택 결과로 생기는 일 → 상위 관리자
현재 선택된 버튼 강조
이전 선택 해제
가격/버프/UI 갱신
무기 선택 상태 저장
-> 이런 건 버튼들의 부모(무기 목록 리스트) 거치기.

예를 들어 이런 건 버튼 안에 넣기 시작하면 좀 위험해져:
public void OnClick()
{
    PlayClickSound();
    Highlight();
    UpgradePanel.UpdateUI();
    PreviousButton.UnHighlight();
    SaveSelectedWeapon();
}
버튼이 점점 "나 눌렸어"를 넘어서 게임 전체 상태를 만지기 시작하거든.
