
내가 어드레서블 쓰는 이유 :

현재 내 게임 구상 : 계속해서 플레이어 데이터가 쌓이는 베이스 씬(상점,가진 아이템 등)에 있다가
전투해서 자원 모으려 갈 때면 매번 초기화되는 또다른 씬으로 이동해서 몬스터 잡고 자원 모아서 
베이스 씬으로 돌아오는 식. 
전투하러 가는 씬도 일정 레벨 이상이 되면 더 어려운 난이도의 씬으로 이동할 거고.

어드레서블의 장점 : 
단순히 "Resources.Load()"의 대체"가 아니라 리소스를 효율적으로 관리하는 시스템

1.필요한 순간에만 로드 가능 (메모리 절약)
기존 방식 :
public GameObject enemyPrefab; ->
씬 시작할 때 참조된 에셋들이 미리 메모리에 올라올 수 있음.

Addressables:
Addressables.LoadAssetAsync<GameObject>("Enemy");
필요할 때만 로드:
게임 시작 → 로비만 메모리 사용
스테이지 입장 → 적/맵 로드
스테이지 종료 → 해제

큰 게임일수록 메모리 차이가 커짐.



2. 비동기 로드가 쉬움
일반 로드:
GameObject obj = Resources.Load<GameObject>("Enemy");
큰 파일이면 프레임 끊김 가능.

Addressables:
Addressables.LoadAssetAsync<GameObject>("Enemy");
백그라운드 로딩 가능:

로딩 화면 표시
진행률 표시
게임 안 멈춤



3. 코드가 경로에 덜 묶임
Resources:
Resources.Load("UI/Menu/Button");
폴더 이동하면 망가짐.

Addressables:
Addressables.LoadAssetAsync("Button");
실제 파일 위치 바꿔도 Address만 유지하면 코드 수정 없음.


4.씬도 Addressables로 관리 가능
Addressables.LoadSceneAsync("Stage1");

장점:
씬을 필요할 때만 로드
여러 씬 Additive 관리 쉬움
대형 맵 스트리밍 가능


5. DLC/패치 시스템 만들기 쉬움
게임 설치 후:
새 캐릭터
새 맵
이벤트 에셋
다운로드 가능.

예:
서버
 ├─ Character_A
 ├─ Character_B
 └─ Event_Map
게임 업데이트 없이 추가 가능.

모바일 게임들이 많이 쓰는 이유 중 하나.

6. 에셋 중복 관리
같은 텍스처를 여러 프리팹이 쓰는 경우:

기존 방식:
MonsterA → Texture
MonsterB → Texture
MonsterC → Texture
잘못 관리하면 중복 로드될 수 있음.

Addressables는 의존성 관리해서:
Texture 1개 로드
공유해줌.


7. 로드/해제 추적 가능
로드:
var handle = Addressables.LoadAssetAsync<GameObject>("Enemy");

해제:
Addressables.Release(handle);
어떤 에셋이 살아있는지 관리하기 쉬움.





내 게임에서의 장점 : 
UI 프리팹 동적 생성
씬 비동기 로드
메모리 관리
